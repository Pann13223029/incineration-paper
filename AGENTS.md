# AGENTS.md

Repository-wide guidance for coding and writing assistants.

## Purpose

This is a paper-first derivative repo seeded from the defended thesis workspace. The task is to develop a journal-style paper without destabilizing the empirical core or confusing it with the original thesis workflow.

## Source Hierarchy

| Need | Canonical place | Rule |
|:--|:--|:--|
| Current sample and model facts | `output/sample_definition.md`, `output/adoption_results.md`, `output/regression_results.md` | Verify here before repeating numbers. |
| Robustness and sensitivity facts | `output/robustness_results.md`, `output/data_quality_sensitivity.md` | Use these for reviewer-risk and caveat language. |
| Claim synchronization | `output/claim_verification.md`, `output/claim_evidence_map.md` | Run `npm run claims:verify` after claim wording changes. |
| Empirical design logic | `code/scripts/panel_utils.py` | Inspect before changing model/sample descriptions. |
| Active paper prose | `paper/manuscript/paper.md` | Main article-writing target. |
| Active LaTeX reading source | `paper/manuscript/paper.tex` | Keep synchronized when changing final PDF wording. |
| Paper-facing evidence copies | `paper/evidence/current/` | Generated by `npm run paper:sync`; do not hand-edit. |
| Thesis baseline | `thesis/thesis.tex` | Reference-only unless explicitly asked to edit thesis. |
| Legacy thesis support | `research/` | Reference-only unless explicitly asked. |

## Core Rules

1. Keep the empirical core intact; paper claims must narrow or explain evidence, not invent new evidence.
2. Treat `paper/manuscript/paper.md` as the active writing target.
3. Treat `output/*` as the source of truth for current results.
4. Treat `paper/evidence/current/*` as generated copies of selected outputs.
5. Treat `thesis/` and `research/` as baseline/legacy material by default.
6. Move defensive detail into the supplement rather than bloating the main article.
7. Do not manually edit generated evidence snapshots, manifests, or submission exports unless the task specifically requires repairing generated output.

## Workflow Gates

Use the smallest sufficient gate:

| Change | Gate |
|:--|:--|
| Paper prose only | Edit `paper/manuscript/paper.md`; rebuild PDF only if needed. |
| New or changed empirical claim | Verify against `output/*`, then run `npm run claims:verify`. |
| Analysis or sample logic change | Run `npm run analysis:rebuild`, `npm run paper:sync`, and `npm run claims:verify`. |
| Paper evidence copy change | Run `npm run paper:sync`, then `npm run paper:check`. |
| Submission artifact refresh | Run `npm run paper:export:nopdf` and `npm run paper:build:latex`. |
| Pre-push check | Run `npm run paper:check`, `npm run claims:verify`, and `git diff --check`. |

## Command Notes

- `npm run analysis:rebuild` is the canonical empirical rebuild path and calls `code/scripts/07_rebuild_analysis.py`.
- `npm run paper:check` verifies that required `output/` sources exist and that `paper/evidence/current/` is synced.
- `npm run paper:export:nopdf` avoids the local Chrome PDF path; use `npm run paper:build:latex` for the authoritative PDF.
- CI also uses the canonical rebuild orchestrator, so do not maintain a separate stage list in documentation.

## Writing Defaults

- Audience: waste management, industrial ecology, and environmental policy readers.
- Main contribution: separating adoption into generation from conditional generator performance in one national facility panel.
- Framing default: selective modernization on the adoption margin plus bounded responsiveness on the efficiency margin.
- Mechanism language: calibrated, not exclusive. Do not claim replacement is uniquely identified.
- Generalization: cautious. Japan is the empirical case, not automatic proof for all waste systems.

## Reviewer-Sensitive Caveats

Keep these visible when discussing robustness:

- Duplicate official facility codes are addressed through `output/data_quality_sensitivity.md`.
- Heating-value noise is treated as a control issue and sensitivity-checked in `output/data_quality_sensitivity.md`.
- `output/claim_verification.md` verifies registered snippets and stale-pattern guards, not every possible manuscript sentence.
- The paper models electricity generation cleanly; heat recovery/district heating is future work unless new data are added.

## Maintenance

- Update `README.md`, `ARCHITECTURE.md`, `AGENTS.md`, and `paper/README.md` together when workflow boundaries change.
- Keep this file free of volatile result tables unless they are needed for source hierarchy.
- If a generated file becomes stale, regenerate it through the appropriate script instead of hand-editing.
