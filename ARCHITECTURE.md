# Paper Repo Architecture

This file defines how the repo is supposed to work. It is intentionally less narrative than `README.md`: use it to decide where a change belongs, what must be rebuilt, and which files are sources of truth.

## Design Goals

1. Preserve one empirical core.
2. Keep the paper layer downstream of generated evidence.
3. Make submission artifacts reproducible without turning them into new evidence.
4. Keep the defended thesis available as reference history, not as the active writing target.
5. Prevent drift between data outputs, paper claims, README-facing summaries, and CI.

## Layer Model

| Layer | Purpose | Paths | Edit rule |
|:--|:--|:--|:--|
| Evidence core | Data parsing, estimation, robustness, sensitivity, and claim verification | `data/`, `code/`, `output/` | Edit only when changing empirical logic or generated outputs. |
| Paper layer | Active journal-style manuscript, supplement, figures, submission package, and paper-facing evidence copies | `paper/` | Edit for article development; keep empirical claims downstream of `output/*`. |
| Reference thesis layer | Defended thesis baseline and legacy review/defense machinery | `thesis/`, `research/` | Reference by default; do not treat as the active paper workspace. |
| Repo guidance layer | Human and assistant navigation, workflow contracts, and architecture | `README.md`, `ARCHITECTURE.md`, `AGENTS.md` | Update when workflow boundaries, source-of-truth rules, or command gates change. |

## Source-Of-Truth Rules

| Claim type | Source of truth | Notes |
|:--|:--|:--|
| Sample sizes and model outputs | `output/*.md`, `output/*.csv`, `output/manifests/*.json` | Generated by scripts; do not hand-edit for convenience. |
| Paper-facing copied evidence | `paper/evidence/current/*` | Generated by `npm run paper:sync`; should match selected `output/*` files. |
| Active paper prose | `paper/manuscript/paper.md` | Edit this first for article prose; keep `paper.tex` synchronized when changing final LaTeX wording. |
| LaTeX reading PDF | `paper/share/waste-management-manuscript-latex.pdf` | Tracked convenience artifact for cross-device reading; regenerated by `npm run paper:build:latex`. |
| Submission package | `paper/submission/*` | Generated or assembled from paper sources; not a source of empirical truth. |
| Thesis baseline | `thesis/thesis.tex` | Defended reference, not the active paper target. |

`code/scripts/07_rebuild_analysis.py` is the canonical stage orchestrator. CI should call it rather than duplicating the script list.

## Command Map

| Command | Scope | Output |
|:--|:--|:--|
| `npm run analysis:rebuild` | Rebuild empirical outputs and claim verification | `data/processed/*`, `output/*` |
| `npm run paper:sync` | Refresh paper-facing evidence copies | `paper/evidence/current/*`, `paper/evidence/README.md` |
| `npm run paper:check` | Check required evidence exists and synced copies are current | Exits nonzero if outputs are missing or copied evidence is stale |
| `npm run claims:verify` | Run registered claim and stale-wording checks | `output/claim_verification.md`, `output/claim_evidence_map.md` |
| `npm run paper:export:nopdf` | Export Markdown, HTML, and DOCX submission files without Chrome PDF | `paper/submission/waste-management-manuscript.{md,html,docx}` |
| `npm run paper:build:latex` | Rebuild figures and the authoritative LaTeX reading PDF | `paper/submission/waste-management-manuscript-latex.pdf`, `paper/share/waste-management-manuscript-latex.pdf` |

## Submission Build Contract

The authoritative reading PDF is the LaTeX build:

- source: `paper/manuscript/paper.tex`
- command: `npm run paper:build:latex`
- tracked share output: `paper/share/waste-management-manuscript-latex.pdf`

The HTML/DOCX export is useful for submission packaging and review workflows:

- source: `paper/manuscript/paper.md`
- command: `npm run paper:export:nopdf`
- outputs: `paper/submission/waste-management-manuscript.md`, `.html`, `.docx`

The Chrome-generated PDF from `paper:export` is not the preferred artifact because it depends on local GUI/browser behavior. Use the LaTeX PDF unless a target system specifically asks for another format.

## CI Verification Contract

GitHub Actions runs the paper verification workflow on push and pull request. The intended contract is:

1. Install Python, Node, and Tectonic.
2. Rebuild empirical outputs through `code/scripts/07_rebuild_analysis.py`.
3. Sync paper-facing evidence.
4. Export cross-platform submission Markdown/HTML without browser PDF.
5. Build the LaTeX manuscript PDF.
6. Check tracked outputs for unexpected drift.
7. Upload verification artifacts.

CI excludes platform-sensitive rendered figure binaries and the tracked share PDF from the diff gate. Those files remain useful convenience artifacts, but exact binary reproducibility can vary by platform.

## Verification Boundary

`output/claim_verification.md` verifies registered high-risk snippets and stale-language guards. It does not prove that every sentence in the paper is correct. When adding new empirical claims, either connect them to an existing generated output or expand the verifier deliberately.

Reviewer-sensitive caveats are currently concentrated in:

- `output/data_quality_sensitivity.md`
- `paper/evidence/current/data_quality_sensitivity.md`
- `paper/supplement/supplement.md`

## Legacy Thesis Boundary

The thesis layer is retained to preserve the defended baseline and the original supervision/defense context. New paper work should not modify `thesis/` or `research/` unless the task explicitly targets legacy thesis materials.

## Maintenance Rules

1. Prefer changing workflow scripts before documenting manual workarounds.
2. If a command list appears in both CI and docs, keep `package.json` as the discoverable interface.
3. If empirical outputs change, update synced evidence and rerun claim verification before editing paper prose.
4. If the paper workflow changes, update `README.md`, `ARCHITECTURE.md`, `AGENTS.md`, and `paper/README.md` in the same change.
5. Keep volatile timestamps, local machine paths, and commit hashes out of generated docs unless they are intentionally part of a release note.
